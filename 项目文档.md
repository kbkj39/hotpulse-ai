# HotPulse AI — 项目文档（单一事实来源）

说明：此文档是本项目开发阶段的唯一真相。所有需求、约定、接口、数据模型、目录结构、开发规范与部署规范，均以本文件为准。后续开发如与本文档冲突，应先修改本文档，再修改代码。

## 项目概述

### 项目目标

- 构建一个面向财经场景的热点信息抓取与智能分析平台，核心为可编排的多 Agent 协作管线（规划、搜索、抓取、分析、聚合）。
- 支持用户通过自然语言直接触发检索与分析（例如“我想查询近期加密货币热点”），由规划 Agent 自动分解任务并发起多实例并行抓取与分析。
- 系统输出基于证据的热点结论：每条结果包含真实性、相关性、重要性评分及可回溯的原始来源与判断依据，支持审计与回溯。
- 作为个人工具，本项目不包含用户注册/登录与权限管理，默认单用户直接使用。

### 目标用户

- 投研人员
- 财经编辑
- 二级市场交易人员
- 风控与策略分析人员
- 对财经热点和投资分析有需求的普通用户

### 产品定位

这是一个以“多 Agent 热点发现 + 信息治理 + 证据驱动智能问答 + 投资辅助分析”为中心的数据产品。

系统既不是单纯的搜索引擎，也不是单纯的聊天机器人；它通过：

- 上游负责自动检索与治理信息（多 Agent 并行检索、并发抓取、结构化清洗）。
- 下游负责基于清洗后数据的解释与分析（真实性判断、相关性与重要性评分、可溯源的回答生成）。

前端将实时展示 Agent 执行进度与结果溯源，用户可查看每次搜索-抓取-分析的执行记录与依据。

## 需求文档

### 核心需求

#### 热点抓取

系统需要支持两种触发方式：

- 用户主动设置热点关键词。
- 用户直接输入自然语言，例如“帮我查一下今天的热点”。

系统接收到需求后，由规划 Agent 完成意图理解与任务分解，再由多个专职 Agent 并行执行，流程如下：

1. 判断用户要关注的主题、时间范围、信息类型。
2. 选择合适的信息源，生成多维度搜索关键词列表。
3. 将搜索与抓取任务分解为若干子任务，分配给并行执行的搜索 Agent 和抓取 Agent。
4. 多个搜索 Agent 并行检索，多个抓取 Agent 并发获取原始页面内容。
5. 分析 Agent 对每条内容评估真实性、相关性与重要性，低于阈值内容直接过滤。
6. 聚合 Agent 汇总结果并去重排序，最终热点列表推送前端展示，全程附带各 Agent 的执行进度状态。

最终结果需要在“热点雷达”或热点列表中展示，且每条结果必须支持点击跳转到原始页面。

#### 智能体对话

系统需要提供财经问答能力，用户可以围绕热点、市场事件、公司公告、财务信息等内容提问。

智能体需要具备以下能力：

- 接收用户自然语言查询，自动触发多 Agent 协作流程，无需用户手动配置抓取任务。
- 结合知识库回答财经问题。
- 结合实时热点信息回答问题。
- 对回答结果保留来源依据。
- 保存历史对话，支持基础上下文延续。
- 在对话界面实时展示当前正在执行的 Agent 步骤，令用户可感知系统进度。

#### 热点信息处理

抓取后的信息不能直接原样展示，必须经过处理：

- 真实性判断
- 相关性过滤
- 去重处理
- 一句话摘要生成
- 标签生成
- 热度评分

处理后的内容才允许进入前端展示层和问答上下文。

#### 多 Agent 协作

本系统的核心执行架构是多 Agent 协作管线。以用户输入"我想查询近期加密货币热点"为例，完整流程如下：

**规划 Agent（Planner）**

- 接收用户自然语言输入，理解查询意图。
- 提取主题（加密货币）、时间范围（近期）、信息类型（热点资讯）。
- 生成结构化任务计划（Task Plan）：候选信息源列表、多维度搜索关键词、各子任务的优先级与并发策略。
- 将子任务分发给下游专职 Agent。

**搜索 Agent（Searcher，多实例并行）**

- 接收规划 Agent 分配的关键词与目标信息源（如 Twitter/X、Reddit、CoinDesk、交易所公告、主流财经媒体等）。
- 多个实例并行执行，各自负责一组信息源，互不阻塞。
- 返回候选 URL 列表及页面摘要片段。

**抓取 Agent（Crawler，多实例并行）**

- 接收候选 URL，并发抓取原始页面内容。
- 处理反爬应对、超时重试与内容提取，输出结构化原始文档。

**分析 Agent（Analyzer）**

- 对每条原始文档独立执行三项评估：
  - **真实性评估**：判断来源可信度，识别明显虚假或误导性内容，输出置信度与判断依据。
  - **相关性评分**：衡量内容与用户原始查询的语义匹配程度（0–1 分值）。
  - **重要性评分**：结合热度、传播范围、时效性、市场影响力等维度综合评分（0–1 分值）。
- 低于预设阈值的内容直接过滤，不进入聚合层。

**聚合 Agent（Aggregator）**

- 汇总所有分析结论，执行跨条目语义去重与多维排序。
- 生成最终热点列表，每条结果包含：标题、来源、发布时间、摘要、真实性/相关性/重要性三项评分、原始链接。

**执行约束**

- 规划 Agent 是流程的唯一入口，负责全局任务调度。
- 各 Agent 之间通过标准化消息结构通信，互相不直接调用。
- 整个流程对用户透明，界面实时展示当前执行阶段（规划中 → 搜索中 → 抓取中 → 分析中 → 完成）。
- 任意单个 Agent 失败仅影响其负责的子任务，不阻断整体流程。

### 扩展需求

#### 热点搜索与展示增强

- 支持按最新时间排序。
- 支持按重要程度排序。
- 支持按相关性排序。

- 支持一句话搜索热点，而不只是关键词配置。

#### 可视化能力

- 展示热点趋势。
- 展示热度统计。
- 支持生成每日财经日报。

#### 智能体扩展能力

- 支持自定义热点监控逻辑。
- 支持多 Skills 链式调用。
- 支持结合知识库和热点信息给出投资分析建议。
- 支持用户在对话中直接追问某条热点的深度分析，自动触发针对性的多 Agent 子流程。
- 后续支持多模态输入输出，例如图表、行情数据、文本混合分析。

### 需求验收标准

#### 热点抓取

- 用户输入关键词或自然语言后，系统能返回有效热点结果。
- 热点结果必须包含标题、来源、发布时间、摘要、原文链接。
- 用户点击后能跳转到原页面。

#### 智能体问答

- 用户提出财经问题后，系统能返回结构化回答。
- 回答应尽量附带来源或依据。
- 系统必须记录对话历史。

#### 内容处理

- 系统必须能够识别低质量或低相关性内容。
- 系统必须支持基础摘要和标签生成。
- 系统必须支持热度评分字段。

#### 多 Agent 协作

- 用户输入自然语言查询后，系统必须自动触发完整的多 Agent 协作流程，无需用户手动配置。
- 规划 Agent 必须输出可追踪的结构化 Task Plan。
- 搜索 Agent 与抓取 Agent 必须支持多实例并行，不得串行阻塞。
- 分析 Agent 必须对每条内容独立输出真实性、相关性、重要性三项评分，并附带评判依据。
- 聚合结果必须按综合评分排序，低于阈值内容不得出现在最终列表中。
- 界面必须实时展示当前 Agent 执行阶段，用户可感知进度。
- 单个 Agent 失败时系统不应整体中断，需进行局部容错处理。

### 设计原则与产品约定

- 本文档是项目唯一真相。
- 所有需求变更必须先更新文档。
- **本项目为个人工具，无需用户注册、登录、权限管理等用户体系，所有功能默认单用户直接可用。**
- 系统优先保证信息质量，其次再追求信息数量。
- 所有 AI 输出都应尽量提供证据来源。
- 所有热点结果必须可回溯到原始来源。
- 多 Agent 执行过程必须可观测，不允许将内部决策过程完全黑盒化。
- Agent 之间通过标准化消息结构交互，不允许直接耦合调用。

### 风险与约束

- 部分信息源可能存在反爬限制，抓取 Agent 需内置重试与降级机制。
- 大模型可能产生幻觉，因此回答必须尽量基于证据；分析 Agent 的结论必须附带依据，不允许仅输出分值。
- 热点系统对实时性要求较高，需要平衡抓取频率与成本。
- 财经信息存在误导性内容，因此真实性校验是刚需，不是可选项。
- 多 Agent 并行执行会显著增加大模型调用量，需合理设计并发上限与 Token 预算控制策略。
- 规划 Agent 的任务分解质量直接决定最终结果质量，需对 Task Plan 输出格式做严格约束与校验。

## 开发文档

### 技术框架

#### 前端技术选型

| 分类 | 选型 | 说明 |
|------|------|------|
| 核心框架 | React 19 + Vite 7 | 业界主流 UI 框架与极速构建工具 |
| 样式方案 | Tailwind CSS 4 | 原子化 CSS，开发效率高 |
| 实时通信（热点推送） | Socket.io-client | 后端推送新热点条目到前端列表 |
| 实时通信（Agent 进度） | 原生 EventSource（SSE） | 订阅 Agent 执行步骤流，展示实时进度 |
| 全局状态 | Zustand | 轻量状态管理，避免 Redux 冗余 |
| HTTP 请求 | Axios | 统一封装，拦截器集中处理错误 |
| 单元测试 | Vitest | Vite 生态原生测试框架 |
| E2E 测试 | Playwright | 覆盖关键用户流程 |

#### 后端技术选型

| 分类 | 选型 | 说明 |
|------|------|------|
| 核心框架 | Java 21 + Spring Boot 3 | LTS 版本，虚拟线程（Project Loom）支持高并发 Agent 并行 |
| Web 层 | Spring Web + Spring Validation | RESTful 接口 + SSE 流式推送（SseEmitter） |
| 持久化 | Spring Data JPA + PostgreSQL 16 | 主流 ORM，关系型存储 |
| 向量存储 | PGVector（PostgreSQL 扩展） | 嵌入式向量检索，复用已有 PostgreSQL 实例 |
| 缓存 | Redis | 热点结果缓存、Agent 执行状态暂存 |
| 消息队列 | RabbitMQ | 基础设施已配置（Exchange/Queue/Binding），当前仅保留备用；定时抓取任务已改为虚拟线程直接调用 AgentOrchestrator |
| 数据库迁移 | Flyway | 所有表结构变更通过 SQL 文件版本管理 |
| AI 接入 | Spring AI 1.1.4 | 统一 ChatClient / EmbeddingClient 抽象层，支持热切换模型；原生 OpenAI Java SDK 集成、原生结构化输出（JSON Schema）支持 |
| Agent 并行 | Java 21 虚拟线程 + CompletableFuture | 多 Searcher / Crawler 并发执行，无需额外线程池配置 |
| 构建工具 | Maven | Java 生态标准构建工具 |

#### AI 与数据采集选型

| 分类 | 选型 | 说明 |
|------|------|------|
| 大模型服务 | OpenRouter API | 统一接入多厂商模型（DeepSeek / Claude / GPT），单一 API Key |
| 模型热切换 | Spring AI 抽象层 | 切换 OpenAI / Ollama / 阿里百炼只需改配置，无需改代码 |
| 金融知识库 | 同花顺问财 SkillHub | 替代本地 RAG；AnalyzerAgent 调用 IwencaiSkillService 获取权威金融证据，用于热点真实性/相关性评估 |
| 静态网页抓取 | Jsoup | 轻量 HTML 解析，适合结构化财经站点 |
| 动态网页抓取 | Playwright for Java | 处理需要 JS 渲染的页面（Twitter/X、部分交易所公告等） |
| RSS/Feed 解析 | Rome（Java RSS 库） | 订阅主流财经媒体 RSS，优先级最高、成本最低 |
| 任务调度 | Spring Scheduler + Quartz | 定时抓取与日报生成 |

#### 可观测性与部署选型

| 分类 | 选型 | 说明 |
|------|------|------|
| 链路追踪 | OpenTelemetry | TraceId 贯穿请求→Agent 步骤→数据库，支持跨 Agent 链路可视化 |
| 监控告警 | Prometheus + Grafana | 抓取成功率、Agent 耗时、Token 消耗等指标 |
| 容器化 | Docker + docker-compose | 本地一键启动全套基础设施 |
| CI/CD | GitHub Actions | 自动化构建、测试、发布 |

```text
    │   │                                    提供 startNewConversation() 清空当前对话开始新一轮
    │   └── agentExecutionStore.ts        ← Zustand store：管理当前 Agent 执行的全局 SSE 状态；
    │                                        start(executionId) 调用 agentSse.subscribe() 建立 EventSource，
    │                                        接收步骤事件更新 steps[]；
    │                                        收到 System DONE/FAILED 后关闭 EventSource 并写入 finalAnswer；
    │                                        SSE 末尾事件中的 hotspots[] 直接写入 hotspotStore.addHotspot()
    │
    └── types/
        ├── hotspot.ts                    ← Hotspot、HotspotFilter、ScoreDetail 接口定义
        ├── chat.ts                       ← Message、Conversation 接口定义
        ├── agent.ts                      ← AgentStepEvent、TaskPlan、AgentStatus 枚举定义
        └── monitorKeyword.ts             ← MonitorKeyword 接口定义（id、keyword、enabled、
                                              crawlIntervalHours、lastCrawledAt、createdAt）
```

#### 文件依赖关系

```
pages → hooks → services → 后端 API
pages → components（纯展示，接收 props）
hooks → store（读写 Zustand 状态）
AgentProgressPanel ← useAgentExecution ← agentExecutionStore ← agentSse.ts ← SSE /api/v1/agent/stream/{id}
MonitorKeywordPanel.KeywordCard ← useAgentExecution（读取 agentExecutionStore 全局状态）
hotspotStore.addHotspot ← useSocket（Socket.IO newHotspot 事件）
hotspotStore.addHotspot ← agentExecutionStore（SSE System.DONE 事件中的 hotspots[]）
HotspotsPage ← useSocket ← socket.ts ← Socket.io /ws/hotspots
```

#### 关键约定

- 所有 HTTP 请求统一通过 `services/api.ts` 发出，禁止在 Hook 或组件内直接 fetch/axios。
- SSE 连接（Agent 进度）统一通过 `services/agentSse.ts` 管理，订阅时传入 `executionId`。
- Socket.io 连接（热点推送）统一通过 `services/socket.ts` 管理。
- 页面组件只负责布局与交互，业务数据全部来自自定义 Hook。
- 组件样式说明：本项目支持 Tailwind CSS，同时部分组件（尤其是开发阶段的快速迭代组件）使用内联 style。
  推荐长期规范：以 Tailwind 为主，必要时可使用受限的内联样式以便快速验证 UI 逻辑。
- 热点卡片必须展示：标题、来源、发布时间、摘要、真实性/相关性/重要性三项评分、原文链接。
- AgentProgressPanel 必须逐步渲染每个步骤，不能等全部完成才显示。

    │   → RagService → ChatClient → 生成 RAG 回答                                 │

#### 数据模型

所有表通过 Flyway SQL 文件管理迁移，必须包含 `created_at` 字段，禁止手动修改线上库结构。

| 表名 | 对应实体 | 说明 | 关键字段 |
|------|---------|------|---------|
| `sources` | `Source` | 信息源注册表 | `name`、`type`（RSS/HTML/API）、`base_url`、`reputation_score` |
| `raw_pages` | `RawPage` | 原始抓取网页 | `source_id`、`url`、`canonical_url`、`fingerprint`（SHA-256）、`status`、`fetched_at` |
| `documents` | `Document` | 清洗后结构化文档 | `raw_page_id`、`title`、`content`、`published_at`、`summary`、`author`、`source_url`、`source_name`（来源名称，入库时写入）、`tags_json`（JSON 格式标签列表，由 TagDocumentSkill 生成） |
| `chunks` | `Chunk` | 文本切片 | `document_id`、`chunk_index`、`text`、`token_count` |
| `embeddings` | — | 向量（PGVector 表） | `chunk_id`、`model`、`embedding vector(1536)` |
| `hotspots` | `Hotspot` | 最终热点条目 | `document_id`、`execution_id`（来源 AgentExecution）、`truth_score`、`relevance_score`、`importance_score`、`hot_score`、`tags`、`analysis_evidence` |
| `conversations` | `Conversation` | 对话会话 | `title`、`created_at` |
| `messages` | `Message` | 对话消息 | `conversation_id`、`role`（user/assistant）、`content`、`sources_json` |
| `agent_executions` | `AgentExecution` | 一次完整 Agent 执行记录 | `query`、`status`（RUNNING/DONE/FAILED）、`task_plan_json`（Planner 输出的 TaskPlan JSON）、`conversation_id`（关联的对话 ID）、`started_at`、`completed_at` |
| `agent_execution_steps` | `AgentExecutionStep` | 单个 Agent 步骤执行记录 | `execution_id`、`agent_name`、`status`、`detail_json`（输入/输出/评分摘要）、`started_at`、`completed_at` |
| `monitor_keywords` | `MonitorKeyword` | 用户设置的监控关键词 | `keyword`（唯一）、`enabled`、`crawl_interval_hours`（定时间隔，null=不定时）、`last_crawled_at`（上次触发时间）、`created_at` |
| `daily_reports` | `DailyReport` | 每日财经日报 | `report_date`（唯一）、`content`（Markdown 全文）、`hotspot_count`、`generated_at` |

**持久化原则：**

- 所有关联关系在 JPA 实体与 DB 约束中同时声明。
- 不允许手动修改线上库结构，必须通过 Flyway SQL 文件管理。
- 高频查询字段必须建立索引：`raw_pages.fingerprint`、`raw_pages.canonical_url`、`documents.published_at`、`hotspots.hot_score`、`hotspots.importance_score`、`messages.conversation_id`、`agent_execution_steps.execution_id`。
- `embeddings` 表使用 PGVector `ivfflat` 索引，维度锁定为 1536。

#### Skills 设计约定

每个 Skill 是独立 Spring Bean，实现 `Skill<I, O>` 接口，返回值统一包含 `status`、`traceId`、`durationMs`、`error`：

```java
public interface Skill<I, O> {
    String name();
    Duration timeout();
    SkillResult<O> execute(I input);
}

public record SkillResult<O>(
    String status,     // "ok" | "error"
    O data,
    String error,
    String traceId,
    long durationMs
) {}
```

| Skill | 调用方 | 输入 | 输出 |
|-------|-------|------|------|
| `PlanSearchSkill` | PlannerAgent | 自然语言查询字符串 | TaskPlanDto（intent、信息源列表、关键词组、优先级） |
| `FetchAndIngestSkill` | CrawlerAgent | URL + sourceId | Document（入库后的结构化文档） |
| `SearchRagSkill` | 保留备用（AnalyzerAgent 已改用 IwencaiSkillService） | 查询字符串、topK | 证据列表（chunk + url + publishedAt + score） |
| `SummarizeSkill` | IngestService、AggregatorAgent | 文档文本 | 一句话摘要字符串 |
| `VerifyTruthfulnessSkill` | AnalyzerAgent | execute()：Document + 证据列表；<br>executeBatch()：List\<Document\> + 证据列表 | execute()：真实性评分（0–1）+ 依据文本；<br>executeBatch()：List\<BatchResult\>{index, score, reason} |
| `TagDocumentSkill` | IngestService | Document 标题 + 正文 | 标签列表（最多 5 个） |
| `GenerateDailyReportSkill` | DailyReportJob | 当日 Hotspot 列表 | 结构化日报（Markdown） |

#### 抓取策略约定

- 抓取优先级：RSS/API → 静态 HTML（Jsoup）→ 动态渲染（Playwright）。
- 同一域名并发限制 ≤2，请求间隔 ≥1s。
- URL 去重基于 `canonical_url`，内容去重基于 SHA-256 指纹（`fingerprint` 字段）。
- 抓取失败：5xx 最多重试 3 次（指数退避）；遭遇反爬时降低该 Source 的 `reputation_score` 并记录日志。
- 默认遵守 `robots.txt`；特殊来源需在 `sources` 表中记录授权标志。

#### 向量化与 RAG 约定

- 切片优先按 HTML DOM 语义段落划分，fallback 为句子级切分；目标 800–1000 token/chunk，重叠 128 token。
- Embedding 模型默认 `text-embedding-3-large`，维度 1536。
- 检索 `top_k = 8`，相似度度量 cosine。
- 所有检索结果必须携带 `source_url` 与 `published_at`，供 AnalyzerAgent 引用与前端展示。

```
DELETE /api/v1/monitor-keywords/{id}  ← 删除关键词
POST /api/v1/monitor-keywords/{id}/trigger ← 立即触发单个关键词抓取，返回 { executionId }
POST /api/v1/monitor-keywords/trigger-all ← 立即触发所有已启用关键词的抓取，返回 [{ keywordId, executionId }]
```

请求/响应说明：
- `executionId` 可用于前端订阅 `/api/v1/agent/stream/{executionId}` SSE，实时查看该抓取任务的 CRAWL TRACE；
- `crawlIntervalHours` 为 null 或 0 表示不定时；定时任务由 `CrawlScheduleJob` 扫描并调用服务端触发接口。

#### 可观测性约定

系统必须暴露以下指标（通过 Spring Actuator + Prometheus）：

| 指标 | 目标阈值 | 说明 |
|------|---------|------|
| 抓取成功率 | > 95% | CrawlerAgent 成功率 |
| 文档入库 p95 延迟 | < 5s | IngestService 完整流程耗时 |
| 向量检索 p95 延迟 | < 200ms | VectorSearchService 查询耗时 |
| Agent 完整执行 p95 | < 30s | AgentOrchestrator 从发起到最终输出热点列表 |
| Planner 单次 p95 | < 5s | PlannerAgent 生成 TaskPlan 耗时 |
| Skill 调用成功率 | > 99% | 所有 Skill 执行 |
| 系统 5xx 错误率 | < 1% | 全局错误率 |
| Token 消耗 / 次执行 | 监控预算 | 超出 `AGENT_TOKEN_BUDGET_PER_EXECUTION` 时告警 |

告警策略：抓取失败率 5 分钟内超过 20% 触发告警；Embedding 服务不可用触发 P1；Token 消耗超预算 2 倍触发告警。

#### 安全与合规约定

- 对所有外部输入（URL、查询词）进行基本合法性校验，防止 SSRF 攻击（URL 白名单机制）。
